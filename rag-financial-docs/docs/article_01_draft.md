# У меня есть документы. С чего начать?

*Статья 1 из серии: RAG для финансовых и юридических документов*

---

Представьте, что вы только что получили задачу: сделать так, чтобы AI мог отвечать на вопросы по документам клиентов — контрактам, финансовым отчётам, внутренним политикам. У вас есть папка с PDF-файлами и желание сделать что-то умное.

Первая мысль звучит примерно так: *"А что если просто скормить всё это ChatGPT?"*

Это разумный вопрос. И ответ на него — хорошая точка входа в тему RAG.

---

## Почему нельзя просто скормить документы в LLM

Самый очевидный подход: взять все документы, вставить их в промпт и попросить AI ответить на вопрос. Проблема в том, что это не работает в реальных условиях.

Во-первых, у языковых моделей есть контекстное окно — ограничение на количество токенов, которое они могут обработать за один вызов. GPT-4o поддерживает ~128 000 токенов. Звучит много, но одна нормальная юридическая база знаний — это сотни документов, каждый по 10–50 страниц. Это миллионы токенов. Всё туда не влезет.

Во-вторых, даже если влезет — дорого и медленно. Каждый вопрос будет стоить как вызов с полным корпусом документов. При реальной нагрузке это убивает экономику.

В-третьих, и это самое важное: модели плохо работают с очень длинным контекстом. Исследования показывают так называемый **"lost in the middle" эффект** — информация из середины длинного контекста теряется. Модель лучше запоминает начало и конец, а всё между ними — хуже.

**RAG решает эту проблему иначе:** не давать модели всё сразу, а сначала найти только то, что нужно для ответа на конкретный вопрос, и подать только это.

---

## Что такое RAG

RAG — Retrieval-Augmented Generation. Архитектура, в которой перед генерацией ответа происходит поиск по базе знаний.

Упрощённая схема работы:

```
Вопрос пользователя
       ↓
  [Retrieval]  →  найти релевантные фрагменты документов
       ↓
  [Augmentation]  →  собрать контекст из найденных фрагментов
       ↓
  [Generation]  →  LLM генерирует ответ на основе контекста
       ↓
  Ответ с источниками
```

Ключевой инсайт: модель не должна знать содержание документов наизусть. Она должна уметь отвечать на вопросы, когда нужная информация у неё перед глазами.

## Бизнес-кейс этого проекта

На протяжении всей серии мы строим конкретную систему: **AI-ассистент для B2B финтех/консалтинговой платформы**.

Когда приходит новый корпоративный клиент, система накапливает документы: финансовые отчёты, контракты, compliance-политики, продуктовую документацию, регуляторные справки. Сотрудники и клиенты должны быстро находить ответы по всей этой базе.

Примеры вопросов, на которые должна отвечать система:

- *"Какова была выручка клиента X в Q3 2023?"*
- *"Нарушает ли этот контракт нашу политику по риску?"*
- *"Какие документы нужны для онбординга клиента из Германии?"*
- *"Выдели ключевые условия NDA с компанией Y"*

Почему именно этот кейс:

- **Несколько типов документов** — PDF, DOCX, таблицы, структурированные и неструктурированные данные
- **Точность критична** — финансовые данные, юридические формулировки. Галлюцинации недопустимы
- **Нужна фильтрация по метаданным** — по клиенту, типу документа, дате, уровню конфиденциальности
- **Естественная прогрессия** от простого к сложному

## Типы документов и почему каждый из них — отдельная боль

Прежде чем писать код, стоит понять, с чем именно мы работаем. Документы бывают очень разные, и парсить их по-разному.

### PDF: текстовые vs. сканы

PDF — самый распространённый формат в корпоративной среде. И самый неудобный для парсинга.

**Текстовые PDF** (сгенерированные программно) содержат встроенный текстовый слой. PyMuPDF (`fitz`) извлекает его быстро и качественно. Проблемы начинаются с версткой: многоколоночные отчёты, хедеры и футеры, таблицы, врезки — всё это нарушает линейный поток текста.

**Сканированные PDF** — изображения, в которых нет текста вообще. Здесь нужен OCR. Это отдельная, более сложная задача, которую мы оставим за скобками в первых статьях (но вернёмся к ней).

### Таблицы в PDF

Финансовые отчёты почти всегда содержат таблицы с цифрами. Если просто извлечь текст, таблица превращается в кашу:

```
# Что видит парсер без специальной обработки:
Revenue 2023 Revenue 2022 Change Operating expenses Net income ...
```

Для таблиц нужна специальная библиотека — `pdfplumber`, которая анализирует геометрию страницы и восстанавливает структуру. Лучшая стратегия: PyMuPDF для основного текста + pdfplumber для таблиц.

### DOCX

Word-документы проще. `python-docx` предоставляет доступ к параграфам, заголовкам, таблицам, стилям. Структура документа сохраняется хорошо. Проблемы возникают с встроенными изображениями, схемами и complex layout-ами.

### Структурированные данные (CSV, Excel)

Финансовые данные часто хранятся в таблицах. Их нельзя чанковать как текст — нужно сохранять контекст строк и столбцов. Это отдельная тема, к которой мы вернёмся.

## Метаданные: то, о чём забывают в начале

Когда система хранит документы от 10 разных клиентов, простого поиска по тексту недостаточно. Если сотрудник спрашивает *"Какова выручка клиента X?"*, система не должна возвращать данные из отчёта клиента Y.

Каждый документ должен нести метаданные:

```python
{
    "client_id": "acme_corp",
    "doc_type": "financial_report",      # financial_report | contract | policy | ...
    "year": 2023,
    "quarter": "Q3",                     # если применимо
    "confidentiality": "internal",       # public | internal | confidential
    "source_file": "acme_corp_2023_annual.pdf",
    "page": 4
}
```

Эти поля позволят потом фильтровать перед поиском: *"искать только по документам клиента Acme Corp, тип — финансовый отчёт, год — 2023"*. Это называется **pre-filtering**, и это один из важнейших механизмов точности в RAG.

Метаданные нужно закладывать в схему с самого начала, иначе потом придётся переиндексировать всё с нуля.

## Синтетические данные: почему и как

Для обучающего проекта нам нужны реалистичные документы. Использовать реальные данные клиентов нельзя по соображениям конфиденциальности. Поэтому мы генерируем синтетические.

Синтетические данные — это не просто *lorem ipsum*. Они должны:
- Иметь правдоподобную структуру (хедер, разделы, футер)
- Содержать реалистичные цифры (выручка, расходы, маржа)
- Соответствовать жанровым конвенциям (финансовый отчёт выглядит как финансовый отчёт)
- Создавать controlled test cases — мы знаем правильные ответы, что позволит оценить качество системы

Процесс: GPT-4 генерирует содержимое → `reportlab` форматирует в PDF с хедерами, футерами, таблицами, номерами страниц.

В этом проекте три вымышленных клиента:
- **Acme Corp** — средний tech-бизнес (финансовые отчёты за 2022–2023, контракт, инвойсы, KYC)
- **Globex Inc** — крупный производственный бизнес (финансовые отчёты, NDA, MSA, due diligence)
- **Initech LLC** — стартап (pitch deck, term sheet, финансовые проекции)

Плюс внутренние документы: Risk Management Policy, KYC/AML Procedures, Data Handling Policy.

## Архитектура: общая картина

Прежде чем погружаться в код, важно увидеть полную картину того, что мы строим. Вся система состоит из двух частей.

**Offline pipeline (происходит один раз, при загрузке документов):**

```
Документы (PDF, DOCX)
        ↓
   [Parsing]      ← извлечь текст, таблицы, метаданные
        ↓
   [Chunking]     ← разбить на фрагменты подходящего размера
        ↓
   [Embedding]    ← преобразовать каждый фрагмент в вектор
        ↓
   [Indexing]     ← сохранить векторы + метаданные в Qdrant
```

**Online pipeline (происходит при каждом вопросе пользователя):**

```
Вопрос пользователя
        ↓
   [Query Embedding]    ← вопрос тоже в вектор
        ↓
   [Retrieval]          ← найти похожие фрагменты (+ фильтрация по метаданным)
        ↓
   [Re-ranking]         ← отранжировать найденное точнее
        ↓
   [Generation]         ← LLM генерирует ответ с контекстом
        ↓
   Ответ + источники
```

В этой статье мы занимаемся только первым шагом offline pipeline: парсинг документов. Остальное — в следующих статьях.

## Код: парсер PDF от нуля

Посмотрим, как выглядит базовый парсер на чистом Python. Я намеренно пишу его так, чтобы было видно каждое решение.

```python
# src/from_scratch/ingestion/parsers/pdf_parser.py

import fitz          # PyMuPDF
import pdfplumber
from pathlib import Path
from dataclasses import dataclass, field


@dataclass
class ParsedDocument:
    """Результат парсинга одного документа."""
    text: str                        # Основной текст
    tables: list[list[list[str]]]    # Список таблиц (каждая — список строк)
    metadata: dict                   # Метаданные: клиент, тип, дата, ...
    pages: int                       # Количество страниц
    source_file: str                 # Путь к исходному файлу


def parse_pdf(path: str | Path, metadata: dict | None = None) -> ParsedDocument:
    """
    Парсит PDF-файл: извлекает текст через PyMuPDF и таблицы через pdfplumber.

    Почему два инструмента?
    - PyMuPDF (fitz): быстрый, хорошо извлекает основной текст,
      сохраняет порядок блоков
    - pdfplumber: медленнее, но умеет восстанавливать структуру таблиц
      (строки, столбцы, ячейки)
    """
    path = Path(path)
    metadata = metadata or {}

    # --- Текст: PyMuPDF ---
    text_parts = []
    doc = fitz.open(str(path))
    num_pages = len(doc)

    for page_num, page in enumerate(doc):
        page_text = page.get_text("text")   # "text" = сохранить порядок блоков
        if page_text.strip():
            text_parts.append(f"[Page {page_num + 1}]\n{page_text}")

    doc.close()
    full_text = "\n\n".join(text_parts)

    # --- Таблицы: pdfplumber ---
    tables = []
    with pdfplumber.open(str(path)) as pdf:
        for page in pdf.pages:
            page_tables = page.extract_tables()
            if page_tables:
                tables.extend(page_tables)

    return ParsedDocument(
        text=full_text,
        tables=tables,
        metadata={
            "source_file": str(path),
            "filename": path.name,
            **metadata,
        },
        pages=num_pages,
        source_file=str(path),
    )
```

Обратите внимание: функция принимает `metadata` как параметр. Метаданные (клиент, тип документа) мы передаём снаружи — парсер сам их не угадывает. Это правильное разделение ответственности.

## Тот же результат с LangChain: 15 строк вместо 80

```python
# src/langchain_impl/ingestion/loaders.py

from langchain_community.document_loaders import PyPDFLoader, UnstructuredPDFLoader
from langchain_core.documents import Document


def load_pdf_langchain(path: str, metadata: dict | None = None) -> list[Document]:
    """
    LangChain-версия парсинга PDF.

    PyPDFLoader — быстрый вариант для текстовых PDF.
    UnstructuredPDFLoader — медленнее, но лучше справляется
    с нестандартной версткой.
    """
    loader = PyPDFLoader(path)
    documents = loader.load()

    # LangChain уже добавляет source и page в metadata.
    # Мы дополняем своими полями.
    if metadata:
        for doc in documents:
            doc.metadata.update(metadata)

    return documents
```

Объект `Document` в LangChain — это `page_content` (текст) + `metadata` (словарь). Стандартный интерфейс работает одинаково для всех типов документов, что очень удобно дальше по pipeline.

## Вердикт по первой теме

Для парсинга LangChain-лоадеры **экономят значительный boilerplate**. `PyPDFLoader`, `Docx2txtLoader`, `DirectoryLoader` — они покрывают 80% случаев.

Кастомный парсер нужен когда:
- Вы хотите по-особому обрабатывать таблицы (например, конвертировать их в markdown-формат для лучшей читаемости)
- Нужна специфическая логика извлечения метаданных (например, вытащить номер контракта из хедера)
- Качество стандартного лоадера не устраивает для конкретного типа документов

**В этом проекте:** используем кастомный парсер для финансовых отчётов с таблицами, LangChain — для всего остального.
